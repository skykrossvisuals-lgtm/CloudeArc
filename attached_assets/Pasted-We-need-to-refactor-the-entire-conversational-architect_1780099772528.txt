We need to refactor the entire conversational architecture so the Workspace and Conversation systems share ONE unified conversational intelligence layer instead of behaving like two disconnected products.

Right now:
- Homepage Conversation mode behaves naturally and collaborates well
- Workspace mode still behaves like an execution-only system
- Workspace sometimes routes casual messages into build execution
- Workspace assistant forgets conversational continuity
- Build prompts can accidentally rerun
- Casual prompts like "Hello" or "I have a brand name" trigger execution behavior instead of collaboration

This must be fixed at the architecture level.

GOAL:
Create a Unified Conversational Intelligence System that powers BOTH:
1. Homepage Conversation
2. Workspace Collaboration

The Workspace should feel like:
- a real collaborator
- aware of context
- aware of project state
- capable of discussing ideas naturally
- capable of switching INTO execution only when execution intent is truly clear

NOT:
- an automated build trigger
- a rigid assistant
- a stateless execution bot

========================================
PHASE 1 — UNIFIED INTENT ARCHITECTURE
========================================

Create a shared conversational engine used by BOTH:
- /conversation
- /workspace

This should replace the duplicated intent systems.

Create:
shared/lib/conversationEngine.ts

This engine should contain:

1. Intent classification
2. Execution confidence scoring
3. Conversation memory
4. Collaboration mode handling
5. Execution gating
6. Build transition detection
7. Rerun protection
8. Conversational continuity

The Workspace MUST consume this shared engine.

NO separate workspace-specific intent logic anymore.

========================================
PHASE 2 — ADVANCED INTENT DETECTION
========================================

Expand intent classification into these modes:

type Intent =
  | "casual"
  | "brainstorm"
  | "advisory"
  | "review"
  | "reflection"
  | "clarification"
  | "planning"
  | "execution"
  | "modify"
  | "debug"

The engine should classify based on:
- wording
- sentence structure
- imperative confidence
- conversational continuity
- project state
- recent assistant/user history

Examples:

"Hello"
→ casual

"I have a brand name"
→ brainstorming / clarification
NOT execution

"What do you think we can add?"
→ advisory / brainstorm

"This feels too empty"
→ review

"Can we improve the dashboard?"
→ planning
NOT execution immediately

"Build the dashboard now"
→ execution

"Implement authentication"
→ execution

"Fix the navbar overflow"
→ debug

========================================
PHASE 3 — EXECUTION GATEKEEPER
========================================

Execution should NEVER happen unless execution confidence passes threshold.

Create:
ExecutionGatekeeper

Rules:
- greetings NEVER execute
- vague ideas NEVER execute
- incomplete ideas NEVER execute
- conceptual discussion NEVER execute
- "I have an idea" NEVER executes
- "should we..." NEVER executes
- "what if..." NEVER executes

Execution only happens when:
- explicit build intent exists
- actionable implementation direction exists
- user clearly wants generation

Before execution:
assistant should naturally confirm understanding conversationally.

Example:
User:
"I want a luxury real estate marketplace"

Assistant:
"That has strong potential. I’m imagining something high-trust and editorial rather than a generic listing platform. Do you want this optimized more for premium visuals, lead generation, or fast property discovery?"

NO BUILD YET.

Only AFTER user confirms:
"Build it"

THEN workspace execution begins.

========================================

The assistant inside workspace should:
- discuss ideas

WITHOUT triggering builds.

The workspace assistant should feel:
- alive
- context-aware
- collaborative
- strategic
- emotionally intelligent in pacing

NOT robotic.

========================================
PHASE 5 — CONVERSATION MEMORY
========================================

Implement persistent conversational memory inside session.

The assistant should remember:
- current idea
- previous suggestions
- product direction
- design taste
- project goals
- discussed tradeoffs

Example:

User:
"I want the UI darker and more cinematic"

Later:
"What else can we improve?"

Assistant:
"The cinematic direction would become much stronger if we reduce flat surfaces and introduce layered lighting around the dashboard panels."

This continuity must work in BOTH:
- homepage conversation
- workspace conversation

========================================
PHASE 6 — BUILD TRANSITION SYSTEM
========================================

When conversation naturally evolves into execution:
the transition should feel organic.

BAD:

GOOD:

Then workspace opens/build starts.

========================================
PHASE 7 — PREVENT PROMPT RERUNS
========================================

Fix rerun behavior completely.

Problems:
- previous execution prompts sometimes replay
- build streams accidentally restart
- stale prompts can re-trigger execution

Implement:
PromptExecutionRegistry

Track:
- active execution ids
- completed prompt hashes
- recent execution timestamps

Prevent:
- duplicate execution within cooldown
- stale replay execution
- accidental stream resurrection
